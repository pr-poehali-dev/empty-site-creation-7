# Журнал: Блокировка не Новых заявок
**Файл журнала:** `src/data/journal/order-lock.md`

---

## 2026-04-30 — обсуждение, до билда

### Контекст задачи
В оптовых заявках (`/admin/orders`) нужно запретить редактирование списка товаров, количества и цены, если заявка имеет статус не «Новая» (`new`). На статусах `confirmed`, `shipped`, `completed`, `archived` редактирование позиций должно быть недоступно. На статусе `new` всё работает как раньше.

### Что выяснили в коде

**Файл:** `src/pages/wholesale-orders/OrderCreatePage.tsx` — одна страница и для создания (`editId === null`), и для редактирования существующей заявки.

**Ключевые точки:**
- `src/pages/wholesale-orders/types.ts:25-31` — статусы хранятся как простые строки: `new | confirmed | shipped | completed | archived`.
- `src/pages/wholesale-orders/OrderCreatePage.tsx:123` — `const [orderStatus, setOrderStatus] = useState("new")`. Для новой заявки по умолчанию `new` — флаг `isLocked` сам собой будет `false`.
- `src/pages/wholesale-orders/OrderCreatePage.tsx:415-453` — useEffect загрузки заявки при редактировании, выставляет `setOrderStatus(data.order.status)`.
- `src/pages/wholesale-orders/OrderCreatePage.tsx:620-623` — хук `useBarcodeScanner({ enabled: !loading, onScan: handleHardwareScan })`.
- `src/pages/wholesale-orders/OrderCreatePage.tsx:818-836` — `handleSave` отправляет PUT (для editId) или POST (для новой) на `ORDERS_URL`.

**Места UI, требующие блокировки:**
- 1078-… — выбор оптовика (customer)
- 1172-1182 — поле поиска товара
- ~режимы поиска (фильтры) — рядом с полем поиска
- 1247-1258 — кнопка «Сканер штрихкодов»
- 1259-1270 — кнопка «Ввод списком» (bulk-paste)
- 1346-1378 — поле ввода штрихкода
- 1503-1508 — крестик удаления строки
- 1518-1524 — инпут количества + кнопки `+`/`−`
- 1534-1540 — инпут цены
- кнопка «Применить ценообразование» (где-то рядом со списком)
- форма создания временного товара (1274-1344) — НЕ блокируем напрямую: она появляется только из дропдауна поиска, а поле поиска уже заблокировано → форма недостижима.
- комментарий — НЕ блокируем (бизнес-требование: можно править на любом статусе).

### Финальное решение (план)

**Один флаг в `OrderCreatePage.tsx`:**
```ts
const isLocked = !!editId && orderStatus !== 'new';
```

**UI-блокировки (только этот файл, только эта страница):**
1. Выбор оптовика — `disabled`
2. Фильтры режимов поиска — `disabled`
3. Поле поиска товара — `disabled`
4. Кнопка «Сканер штрихкодов» — `disabled`
5. Кнопка «Ввод списком» (bulk-paste) — `disabled`
6. Инпут количества — `disabled`
7. Кнопки `+` / `−` у количества — `disabled`
8. Инпут цены — `disabled`
9. Крестик удаления строки — скрыть (`{!isLocked && ...}`)
10. Кнопка «Применить ценообразование» — `disabled`

**Сканер штрихкодов:**
- `useBarcodeScanner({ enabled: !loading && !isLocked, onScan: handleHardwareScan })`
- Отключение действует только на текущей странице с конкретной не-Новой заявкой. На других страницах (приёмка, инвентаризация и т.д.) сканер не затрагивается. При размонтировании страницы хук очищается сам.

**Баннер сверху над списком позиций (только при `isLocked`):**
> Заявка в статусе "{статус}". Редактирование товаров недоступно. Комментарий можно менять.

**Что НЕ трогаем:**
- Поле комментария — редактируемо на любом статусе
- Кнопки смены статуса
- Сохранение / печать / просмотр
- Бэкенд (`backend/wholesale-orders/index.py`) — без изменений
- Миграции БД — не нужны
- `OrderBulkPastePage.tsx` — не трогаем (попасть на неё через заблокированную кнопку «Ввод списком» нельзя)

### Почему такой подход (и почему отказались от других)

Изначально план включал:
- Защиту на бэке (игнорировать `items` при PUT для не-new) — отказались: бэк не трогаем, фронтовой блокировки достаточно для бизнес-задачи.
- Откат статуса «вернуть в Новую» — пока не делаем (не было в требованиях).
- Очистку черновика (draft) при `isLocked` — не нужно: пользователь физически не сможет применить черновик через заблокированный UI.
- Блокировку формы создания временного товара — не нужно: форма достижима только через поле поиска, которое уже заблокировано.

### Принятые ограничения (известные)
- Через DevTools технически можно снять `disabled` и отправить PUT — допустимо, не наш кейс.
- Если менеджер ошибочно перевёл заявку в `confirmed`, откатить нельзя → придётся пересоздавать. Решили оставить как есть, при необходимости добавим откат отдельной задачей.
