# Осталось два этапа 3,4

План реструктуризации работы с заявками. Этапы 1 и 2 уже выполнены (инкрементальное сохранение в БД и защита по версии через `updated_at`). Осталось два этапа.

---

## Этап 3 — Polling между вкладками

### Что делаем
1. Новый лёгкий backend-endpoint `get_version` — возвращает только `updated_at` заявки (без позиций, шапки и пр.)
2. На странице заявки фронт раз в 2–3 сек (активная вкладка) / раз в 30 сек (скрытая, через `document.visibilityState`) дёргает `get_version`
3. Если серверный `updated_at` новее локального — фронт подтягивает полную заявку и обновляет UI без F5
4. Поле в фокусе не перетирается: если пользователь редактирует ячейку — обновление этой строки откладывается до blur
5. Список заявок — отдельный polling раз в 5–10 сек, тоже по версии (max `updated_at` по списку)

### К чему приведёт
- Два менеджера в одной заявке видят изменения друг друга через 2–3 сек без перезагрузки
- Список заявок сам обновляется — новые заявки и смены статусов появляются без F5
- Нагрузка минимальная: `get_version` отдаёт ~50 байт
- Конфликты по `updated_at` (этап 2) почти исчезнут — фронт всегда близок к актуальной версии
- Ввод в поле не сбивается чужими обновлениями

---

## Этап 4 — Информирование о других пользователях

### Что делаем
1. Новая таблица `order_presence` (`order_id`, `user_id`, `user_name`, `last_seen_at`)
2. Фронт на странице заявки раз в 10–15 сек шлёт heartbeat → бэкенд апдейтит `last_seen_at`
3. `get_version` (из этапа 3) дополнительно возвращает список присутствующих (`last_seen_at` < 30 сек назад, кроме себя)
4. Плашка вверху заявки: «Здесь также: Петя, Маша»
5. Модалка конфликта поля: если при сохранении значения сервер видит, что другой юзер только что менял это же поле → возвращает спец-ошибку с чужим значением. Фронт показывает: «Петя поменял на 1500. Ваше 1400. [Сохранить моё / Принять Петино / Отмена]»
6. При уходе со страницы (unmount / `beforeunload`) — запись из presence удаляется

### К чему приведёт
- Менеджеры видят, кто ещё в заявке — меньше случайных пересечений
- При одновременной правке одного поля никто не теряет данные молча
- Никаких блокировок — работа не тормозится
- Нагрузка низкая: heartbeat раз в 10–15 сек, presence пишется в одну строку
